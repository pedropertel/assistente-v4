- Validado por Pedro: continua aparecendo `Hello World · Assistente`, `Banco: conectado` (vindo via novo módulo), botão de tema funciona, console do Safari sem erros.

### Decisões tomadas durante a execução
- **UMD do Supabase mantido como script clássico** (não foi pra um module via `import` do CDN). Razão: o SDK distribui um build UMD que polui `window.supabase` — usar via `window.supabase` no `js/core/supabase.js` é o padrão do projeto descrito no CLAUDE.md (REGRA 5 mostra exatamente `supabaseJs.createClient`). Importar via ESM seria um caminho diferente (ex: `import { createClient } from 'https://esm.sh/...'`) mas exigiria refatorar a convenção. Mantive o caminho descrito no contrato.
- **Ordem de carga garantida pelo navegador:** UMD é blocking (executa síncrono no parse do HTML), `type="module"` tem defer implícito (executa após DOMContentLoaded). Logo, `window.supabase` SEMPRE existe quando `js/core/supabase.js` resolve. Sem race condition.
- **`js/modules/.gitkeep`** criado pra que o git preserve a pasta vazia. Sem isso, ela sumiria no commit (git ignora diretórios vazios). Vai ser populada nas Tarefas 1.7+ (router e os módulos de domínio).
- **Refatoração autorizada:** a palavra "refactor" no commit message (`refactor: extrai cliente supabase pra js/core/supabase.js`) é coerente com REGRA 2 (proibido refatorar sem permissão) — Pedro autorizou explicitamente no prompt da tarefa.

**Status:** ✅ Aprovada em preview (login real funcionou, console mostrou `login ok` + `{user, session}`) e mergeada pra `main` na mesma sessão.

### O que foi feito

  - `Network request failed` / `failed to fetch` → "Sem conexão. Verifique sua internet."
- Window Bridge: criada seção comentada `// WINDOW BRIDGE — funções chamadas por onclick no HTML` no fim do `app.js`, com `window.signIn = signIn` (REGRA 4 do CLAUDE.md).

### Decisões tomadas durante a execução
- **Sem `.trim()` nos campos** — fiel ao prompt do Pedro. Se o usuário digitar email com espaço acidental, o Supabase devolve erro genérico ("Invalid login credentials") e cai no caminho normal de erro. Pode ser refinamento futuro se Pedro reclamar; não criei `Ideias para Revisitar.md` por isso.

- **Fallback final genérico** ("Erro desconhecido ao tentar entrar.") só é usado se o `error` chegar sem `.message` — improvável mas não-zero. Melhor mostrar algo do que mostrar string vazia em vermelho.
- **Onkeydown inline em vez de addEventListener** — alinhado com o padrão Window Bridge do CLAUDE.md (REGRA 4): tudo que dispara JS via interação do usuário fica como atributo HTML, com a função exposta em `window`. Isso mantém o "contrato HTML → window" explícito e auditável; não é elegância JS pura, é disciplina pra evitar cliques mortos.
- **`signIn()` exposta no `window` mesmo dentro de ES module** — sem isso, `onclick="signIn()"` não acharia a função (módulos têm escopo isolado). Padrão do projeto.

## 2026-04-24 — Tarefa 1.5: `onAuthStateChange` + troca de tela login/app

**Status:** ✅ Aprovada em preview (ciclo completo testado em aba normal e privada) e mergeada pra `main` na mesma sessão. **Marca o fim da fase de autenticação** — login/logout funcionando completo.

### O que foi feito
- `js/app.js` reorganizado (refactor autorizado pelo escopo): handler único `onAuthStateChange` registrado no top-level do módulo, com flag `appInitialized` evitando dupla inicialização (REGRA 6 do CLAUDE.md).
- Eventos tratados:
  - `INITIAL_SESSION` ou `SIGNED_IN` com sessão → primeira vez chama `initApp(session)`, próximas (mesma sessão) ignora pra não duplicar inicialização.
  - `SIGNED_OUT` → reseta flag `appInitialized = false` (pra permitir re-init em login subsequente) e chama `showLogin()`.
- `showLogin()` e `showApp()` controlam `display` dos dois screens (`flex` / `none`).
- `initApp(session)` recebe a sessão, mostra o app e roda a query `teste.msg` (a lógica antes solta agora vive aqui dentro).

- `index.html`: `<h1>` + `<div id="status">` encapsulados em `<div id="app-screen" style="display: none;">`, com botão `<button id="logout-btn" onclick="signOut()">Sair</button>` adicionado.

### Decisões tomadas durante a execução
- **`onAuthStateChange` registrado no top-level do módulo, fora de função** — os ES modules carregam uma única vez, então o handler é registrado uma única vez. Não preciso de subscription cleanup nem de `unsubscribe` (a página viva = handler vivo). Se um dia houver SPA com hot reload, aí sim precisaria — não é o caso.
- **`appInitialized = false` no `SIGNED_OUT`** é crítico. Sem isso, se Pedro logar, deslogar, e logar de novo na mesma carga de página, a flag continuaria `true` da primeira vez e `initApp()` não rodaria de novo — tela do app apareceria SEM os dados carregados. Validei mentalmente esse cenário e Pedro confirmou no teste (logou → saiu → logou de novo).
- **`signOut()` NÃO chama `showLogin()` direto** — o `onAuthStateChange` capta o `SIGNED_OUT` e faz isso. Se eu chamasse nos dois lugares, daria pra "funcionar" (idempotente), mas viola o princípio de "uma fonte de verdade" (REGRA 3).
- **`alert()` no erro de signOut** — feio, mas é fallback raro (só roda se a chamada de logout falhar de verdade, ex.: rede caiu). Refinar pra `showToast` quando a Tarefa 1.8 (toast) chegar.
- **Sessão persiste via `localStorage` do Supabase SDK** — comportamento default. Por isso aba normal abre direto no app (`INITIAL_SESSION` vem com sessão restaurada). Isso é o que queremos pra UX mobile do Pedro (não fazer login toda vez que abre o app).
- **`initApp(session)` recebe `session` como parâmetro** mesmo só usando `session.user.email` no log. Preparação pra próximas tarefas — pode precisar passar o user pra módulos. Custo zero, valor futuro.

Valida na prática: logout funcionou em desktop e mobile.

Como sidebar e backdrop são `position: fixed` (fora do flow), sobra só `#main-content` como filho do flex — ocupa 100% sem problema visual. Funciona, mas é um acaso feliz do layout, não uma decisão consciente do template.
